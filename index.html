04 — Pain Points

七大痛点
为什么这件事必须做

以下痛点全部来自一线，不是我们关在办公室想出来的。每一条后面都跟着我们想用 AI 怎么解决。

PAIN · 01
信息获取效率低
手动在 Google、LinkedIn、Apollo、展会名录等平台搜索客户，耗时耗力且覆盖面有限。
一个业务员一天最多覆盖 30-50 家公司，绝大部分是无效翻找。
PAIN · 02
线索处理成本高
手动提取、整理、验证邮箱地址，过程繁琐且易出错。即使用 Snovio 等工具，仍需大量人工补全。
10 个邮箱里 3 个失效、2 个不是决策人，剩下 5 个里只有 1 个值得跟。
PAIN · 03
沟通触达效果差
通用邮件模板群发，缺乏针对性和个性化，客户回复率极低；个性化又写不过来。
行业群发回复率 < 1%，伪个性化（变量替换）回复率 ≈ 1.5%，真个性化能到 8-15%。
PAIN · 04
客户筛选难度大
缺乏精准画像工具，无法有效识别高意向客户，所有线索一视同仁，精力浪费严重。
业务员 80% 时间花在 20% 不会成交的客户身上，剩下高价值客户跟得不够紧。
PAIN · 05
市场洞察能力弱
缺乏系统性工具，难以高效分析全球市场趋势、政策变化、竞品动态，决策靠"听说"。
行业大事只有事后才知道，错过窗口期才知道有过窗口期。
PAIN · 06
回复响应不及时
客户询盘出现在凌晨 3 点（时差），业务员第二天才回复，热度已经凉。多渠道询盘分散，容易漏。
客户的"高峰兴趣窗口"通常只有 6-12 小时，过了就要重新唤醒。
PAIN · 07
域名 / 邮箱预热痛苦
自己群发容易被归类垃圾邮件、域名降级直至废弃；预热又需要持续模拟"真人发送"，无人专门干这件事。
一个废弃的发件域名 = 半年的精力 + 三个月的投递信誉积累全部归零。
05 — Competitive Landscape

三类同行参照
全景剖析与制胜逻辑

市场上对"AI 外贸"赛道发起冲击的，是三类完全不同形态的对手：服务型（苏杭这样的人力外包公司）、工具型（龙虾、Hermes 这样的国内 AI 外贸 SaaS）、通用型（Clay、Smartlead、Apollo 这样的海外冷邮件平台）。 我们要赢，必须搞清楚每一类对手卖的是什么、卡在哪里、为什么外贸老板最终会选我们。

5.1 第一类 · 服务型对手 — 苏杭 Solar

苏杭 Solar（suhang-corp.com）是巴基斯坦的一家数字营销外包公司，2018 年起做营销服务。把它列出来不是为了贬低 — 恰恰相反，他们的服务清单就是市场需求清单。

苏杭的服务清单（来自其 PPT）

自 2018 年起做市场营销外包，主营帮客户提升网络知名度、吸引新客户、整合在线资源。其服务覆盖：

社交媒体运营

Facebook、LinkedIn、Instagram、WhatsApp 多平台运营，制作宣传页+视频，与潜在买家建立联系

付费广告投放

基于位置和行业信息精确锁定，多平台广告投放，定向推广

新客开发 + 寻找客户

利用全球进口商/贸易商/零售商/分销商数据库，多渠道触达，电话+WhatsApp 直接联系

海外参展、邀请买家、列出名单、安排会议、提供本地办事处

5.2 第二类 · 工具型对手 — 龙虾 / Hermes / 通用 AI 外贸 SaaS

除了"人力外包"那一类，市场上还有一大批"工具型 AI SaaS" — 龙虾AI、Hermes AI、Smartlead、Lemlist、Apollo 都属于这一类。 它们是我们更直接的竞争者，也是我们最需要回答清楚"为什么客户最终选我们"的对照组。

这些工具卖什么

— 国内代表

龙虾AI 外贸

国内主流"AI 外贸客户开发系统"。主打：客户挖掘 + AI 开发信生成 + 邮件群发 + 跟进管理。 典型定价 ¥5K-20K/账号/年。面向中小外贸公司业务员自助使用。

— 国内代表

Hermes AI 外贸

AI 外贸营销助手。主打：客户开发 + AI 邮件 + 社媒触达 + 轻量 CRM。 典型功能型 SaaS 形态，业务员需要自己学操作、自己跑流程、自己分析数据。

— 海外代表

Smartlead / Lemlist / Apollo 等

海外冷邮件 + 自动化跟进工具。模板能力强，但对中国出口语境不熟，缺乏本土化（中文支持、对国内业务员习惯、对储能/光伏等行业的深度）。

这一类工具的 5 个共同短板

  • "功能拼盘" vs "Agent 协同" — 它们是把"搜客户 / 写信 / 发邮件 / 跟进 / CRM"5 个独立功能放在一个工具里让用户自己串；我们是 6 个 Agent 在 Memory + Learning 层下自动协同，业务员不用学怎么"串"。
  • "业务员要学" vs "结果直接交付" — 它们是工具，业务员要自己学使用、自己写 Prompt、自己审核每一步、自己看数据；我们是引擎，业务员只在关键节点（报价、谈判、拍板）介入，其余全自动。
  • "通用模板" vs "垂直行业深度" — 它们对所有行业一视同仁（卖钢琴、卖家具、卖光伏共用同一套 Prompt）；我们专做储能/光伏，沉淀行业 know-how（认证标准、HS 编码、各国补贴政策、采购旺季）。
  • "静态产品" vs "持续学习" — 它们的 Prompt 库一年更新一次（产品经理改的）；我们的 Prompt 库每天进化（A/B 真实数据驱动），且专属于客户自己的产品和客群，不和别人共享。
  • "按席位收费" vs "按结果分成" — 它们卖席位订阅（不管你成不成交，年费照交）；我们底费低 + 真见到客户/真成交才有大头收入，风险跟客户对齐。

5.4 为什么外贸人会选我们 · 5 个真实理由

这是销售时反复要回答的核心问题。每一条都基于真实场景，不是营销话术。

— 理由 01 / 已经踩过坑

"我用过龙虾 / Hermes，回复率还是 1%"

国内外贸公司多数已经买过 1-2 个 AI 工具。结果是开发信回复率仍 1-2%、群发被打成垃圾邮件、域名一旦废掉重建要 3-6 个月。 他们要的不是"再多一个工具"，而是真能把回复率拉到 5%+的解决方案 — 这正是我们 Writer Agent + 多渠道编排 + 预热池守护要解决的命题。

— 理由 02 / 没空学新工具

"我每天 10 小时已经塞满，再学一个工具？"

一线业务员的时间表早已饱和。龙虾 / Hermes 是工具型产品 — 业务员要自己写 Prompt、自己审核每一步、自己分析数据，等于"再多一份兼职"。 我们是AI 助理型 — 业务员只在关键节点（报价、谈判）介入，其他全交给 AI。等于"多了一个不要工资的助理"。

— 理由 03 / 行业深度

"通用工具回答不了我的行业问题"

一个储能业务员问"这个客户应该用什么 HS 编码 + 巴西今年储能补贴政策怎样 + 这家公司的采购旺季是几月"， 通用工具一概不知。我们专做储能/光伏，从产品认证（CE / UL / IEC）到关税到典型采购周期都内置进系统。 深度让客户觉得"这个工具懂我的活儿"。

— 理由 04 / 风险对齐

"我愿意按结果付费，不愿意买席位"

中小外贸老板最怕"花了订阅费但没出效果"。 龙虾年费 ¥10K-20K 不管效果好坏都收。我们底费低（覆盖成本）+ 真见到客户 / 真成交了再分成 — 风险全在我们身上，老板敢于试。 只要我们对自己产品有信心，这种收费方式比工具型高出一个量级的获客效率。

— 理由 05 / 信任来源

"做外贸的人做的工具，比做工具的人卖给外贸的人更可信"

龙虾 / Hermes 是产品公司 — 他们没有自己跑外贸业务，所有"成功案例"都是客户单方面提供的。 我们自己就在做储能外贸，有真实订单、真实回复率、真实成交数据。 外贸老板更信"做这行的人做的工具"。这是别人花钱也补不齐的信任壁垒。

— 一句话总结

不是"更便宜的龙虾"
是"完全不同的产品形态"

这些工具不是"差劲的产品"，而是"上一代的产品形态"

龙虾、Hermes、Smartlead 解决了"业务员从 0 到 1 用上 AI"的问题，这一阶段对行业是必要的。 我们做的是下一代 — 把 AI 从"辅助工具"升级到"协同代理（Agentic AI Team）"。 他们的现有客户里大约 30-40% 会发现"工具型不够用"，这部分人就是我们 Phase 2-3 的精准转化对象 — 我们不需要从 0 教育市场，市场已经被前辈预热好了。

知识中枢 · Knowledge Brain（基于 GBRAIN + LLM-WIKI 模式）

6 个 Agent 不是孤立运行 — 他们共享一个会自我增长的"大脑"。这是真正的护城河：跑半年后我们积累的客户档案、决策人画像、销售剧本库，是后来者无论用多少钱也复制不出来的。

— 技术选型决策 —

不自造轮子，直接采用 GBRAIN（MIT 开源）

2026 年 4 月，Andrej Karpathy（前 OpenAI / Tesla AI 总监）提出 LLM-WIKI 模式（GitHub Gist，4 月）—— 用 LLM 持续维护一个 markdown 知识库，知识"compound"而非"重新发现"。 紧接着 4 月 9 日，Garry Tan（Y Combinator 主席）开源了 GBRAIN（MIT 协议）—— LLM-WIKI 模式的生产级实现。

Garry 自己生产环境跑着 17,888 页 / 4,383 人 / 723 公司 / 21 个 cron job，评测 R@5 达 97.9%，比纯向量 RAG 高 31 个点。 几乎为我们这种 "AI Agent 编队 + 客户档案沉淀" 的场景量身设计 —— 直接采用，不重造。

— 三层架构

Brain Repo + Engine + Skills

① Brain Repo：git 管理的 markdown 文件（人类永远可读可编辑）。
② Brain Engine：内嵌 PGLite + pgvector，2 秒就绪，无需 Docker。
③ Skills：29 个开箱即用的 AI 技能脚本（query / ingest / enrich / lint）。

— 三种检索

RRF 混合搜索

HNSW 向量（语义相似）+ Postgres tsvector（关键词精准）+ 知识图谱遍历（关系查询），三路并行 + Reciprocal Rank Fusion。 一次 query 同时搜得到 "南亚储能 EPC" 和 "Ali Khan 的客户都谈过什么"。

— 自动维护

21 个 cron job · 24/7

高频（10-30 分钟）扫邮件 / 社媒；中频（1-3 小时）跑信号雷达；每日生成早报；每周 lint 检查矛盾、过时、孤儿页面、缺失链接。大脑在我们睡觉时自己变聪明。

Brain vs Memory 的纪律 · 这条线不能混

GBRAIN 给我们最重要的纪律是区分两层信息：什么进 Brain（持久），什么留在 Agent Memory（运行时）。一旦混了，Brain 会变成垃圾池。

类别 进 Brain（持久 · GBRAIN） 留 Memory（运行时 · Agent 内存）
客户事实 客户公司业务、规模、决策链、采购周期 本轮对话中临时引用的最新数据
决策人画像 沟通风格、技术偏好、动机、关系状态 本次邮件起草的 prompt 上下文
行业知识 HS 编码、各国补贴政策、认证要求 本次查询临时缓存的 API 响应
销售剧本 沉淀过的"国家 × 行业 × 角色"剧本（赢/丢单复盘） 本次 A/B 实验的中间状态
Deal 状态 Deal 阶段、Stakeholder Map、Risk、Next Action 当前正在生成的 follow-up 草稿
— 为什么这是真正的护城河 —

知识每天 compound · 后来者再有钱也追不上

通用 AI 工具（龙虾 / Hermes）的 Prompt 库是所有客户共享的，不会针对单一客户进化； 我们的 Brain 是每个客户专属的知识中枢，跑 6 个月后，里面有这家客户的 1000 个客户档案、500 个决策人画像、80 份销售剧本 —— 这些数据用钱买不到，用人堆不出来。 这就是 GBRAIN 给我们的最大礼物：一个会复利增长的资产，而不是一个一次性使用的工具。

11 — Tech Stack

技术栈
前沿配前沿，但不为前沿而前沿

每一个选型背后都有一个理由。能用现成 API 不自建，能用托管服务不运维，能用最强 LLM 写信就不省。

— AGENT 编排
Claude Agent SDK / LangGraph
多 Agent 协同最关键的一层。Claude Agent SDK 原生支持多 Agent，LangGraph 灵活度高。先用 Claude Agent SDK，复杂工作流再补 LangGraph。
— LLM 路由
Opus 4.7 + Sonnet 4.6 + Haiku
Writer / Closer 用 Opus（开发信和谈判是命脉，省不得）；Profiler / Strategy 用 Sonnet；批量打分 / 简单分类用 Haiku。
— 浏览器自动化
Playwright + 反检测
LinkedIn / Facebook 动作必须模拟真实浏览器。Playwright + stealth 插件 + 代理池，避开自动化检测。
— 数据源
Apollo + Crunchbase + Clay
不自建 LinkedIn 爬虫（合规风险高）。Apollo 拿决策人，Crunchbase 拿融资动态，Clay 做 enrichment。
— 邮件发送
Mailgun + 自建预热池
Mailgun 性价比 + Webhook 完善。预热池自建（不依赖第三方 warm-up 服务），可控性更高。SPF/DKIM/DMARC 必配。
— 邮箱验证
ZeroBounce
不自造轮子。ZeroBounce $0.008/封，比自建便宜+准。失效邮箱直接丢，避免 bounce 拉低投递信誉。
★ — 知识中枢（核心选型）
GBRAIN（MIT 开源）
Garry Tan（YC 主席）开源的生产级 LLM-WIKI 实现。直接覆盖"向量库 + 关系图谱 + 知识库"三件事。Karpathy 的 LLM-WIKI 是设计模式，GBRAIN 是参考实现。R@5 97.9%，比纯向量 RAG 高 31 点。
— Brain 底层数据库
PGLite + pgvector（GBRAIN 内置）
PGLite 是 WASM 内嵌 Postgres，2 秒就绪，无 Docker 无连接串。MVP 阶段够用，10 万级客户后再迁正式 Postgres（GBRAIN 文档有迁移路径）。
— 知识图谱
GBRAIN 自动抽取（无需 Neo4j）
每次写页面自动抽实体 + 类型边（works_at / advises / supplied_by），零 LLM 调用。比 Neo4j 轻 100 倍。Phase 3 真正大规模再考虑切 Neo4j。
— 后端
Python (FastAPI) + Node (Next.js API)
AI / Agent 用 Python（生态最强）；Web API 和实时数据用 Node。两套之间用 REST + Webhook 通信。
— 前端
Next.js + Tailwind + shadcn/ui
沿用 linqi-landing 经验。仪表盘用 React + WebSocket 推送实时数据。
— 基础设施
Vercel + Railway + Cloudflare
前端 Vercel；后端 Python / 爬虫 Railway 或 Fly.io；Cloudflare 做 CDN + 代理池。早期不上 K8s。
— 数据库 / 文件
Supabase + S3 兼容存储
Supabase 给 Auth + Postgres + Realtime 三件套，省心。文件用 R2（Cloudflare）替代 S3，免出口流量费。
— 监控 / 日志
Sentry + Logfire / Helicone
Sentry 抓异常；Helicone 监控 LLM 调用成本和延迟，不让 token 烧失控。
— CI/CD
GitHub Actions + Vercel/Railway 自动部署
push 到 main 自动跑测试 + 部署。preview branch 给团队 review。
— 协作 / 文档
Linear + Notion + GitHub
Linear 排任务（轻量）、Notion 写文档、GitHub 写代码。三个工具足够，不上 Jira。
— 选型铁律 —

三件事坚决不自造

① 邮箱验证 — 用 ZeroBounce（$0.008/封，自造性价比极低）
② 邮件发送基础设施 — 用 Mailgun（投递信誉是几年沉淀的，不是钱能买到的）
③ LinkedIn 数据 — 用 Apollo API（自爬合规风险高，封号成本高于订阅费）

账面思考（自用阶段，假设的内部成本）

跑完整 1000 个客户的全闭环，预估直接成本：

  • LLM 调用：Opus 写 1000 封信 + Sonnet 1000 份画像 + Haiku 5000 次打分 ≈ $300-500
  • Apollo 订阅：Pro 套餐 ≈ $99/月
  • 邮箱验证（ZeroBounce）：1000 封 ≈ $8
  • Mailgun 发送：5000 封（含跟进）≈ $15
  • Vercel + Railway + Supabase 基础设施：≈ $80-150/月
  • 合计：1000 客户全闭环 ≈ $500-800，单客户边际成本 $0.5-0.8。

对比苏杭模式：同样 1000 个客户，靠人力做下来需要 5-10 人 × 一个月，成本至少 $5000+。 AI 路径的边际成本是其 1/10，规模化后差距更大。

所有"为什么我们这样选"的决策都要写下来。半年后回头看，知道当初是为什么 — 也才知道当初的假设是不是过期了，要不要推翻重来。

每条决策记录的格式

  1. Context — 在什么背景下做这个决策
  2. Options — 当时考虑过哪些选项
  3. Decision — 最终选了哪个
  4. Why — 选这个的核心理由（不超过 3 条）
  5. Trade-offs — 我们放弃了什么
  6. Trigger to revisit — 什么情况下要回来重新审视
16 — FAQ

预想问题
和我们的真实回答

团队、朋友、第一批客户最可能问的问题。先在内部对齐答案，遇到时不慌。

这和 11x.ai / Clay / Smartlead 有啥区别？
三件事：① 我们垂直深耕储能 / 光伏外贸，他们是通用型，对中国供应商→海外采购方的语境理解空白； ② 我们做多 Agent 协同闭环，他们是单点工具拼合； ③ 我们做数字孪生客户模拟器等差异化模块，他们没做。
AI 写的信不会被识别成 AI 进垃圾箱吗？
会，如果直接用 ChatGPT 默认输出。我们的 Writer Agent 内置"刻意降级"规则 — 加入业务员个人风格、口语化片段、合理拼写习惯，并通过 A/B 持续优化。 北极星护栏指标里就有"AI 痕迹识别率 ≤ 20%"的盲测，达不到就不上量。
域名 / 邮箱被封怎么办？
预热池策略 + 多域名轮换 + Sender Score 持续监控。 即使其中一个域名废了，也有备用域名顶上。这是我们 Week 7-8 的核心攻关，做不通后面全部白搭。
不做投资人融资，钱够用吗？
MVP 阶段烧的主要是 LLM token + 几个 SaaS 订阅（Apollo + Mailgun + 基础设施）， 预估 3 个月直接成本 < $5K，比雇一个全职销售还便宜。 Phase 2 内测客户开始反哺现金流后，自循环就能起来。 我们用储能业务的现金流养这个项目，不开外部投资人的天窗。
为什么选储能 / 光伏，不选更大的赛道？
三个原因：① 我们自己在做，有真实业务跑 dogfood，这是最强护城河； ② 储能 / 光伏全球需求在涨，TAM 不算小； ③ 行业够垂直，能积累出"通用 AI BDR 工具"做不到的深度（产品认证、HS 编码、各国补贴政策）。 跑通后再横向扩到光伏、新能源车、储能逆变器等相邻品类。
客户为什么会信任 AI 替他成交？
早期不信任。所以 MVP 阶段所有触达都是"AI 起草 → 人工审核 → 一键发送"，人始终在回路里。 客户看到效果（约到会议、回复率提升）后，逐步放权给 AI 自动跑。 这是一个"信任曲线"，不是一次性买断。
21 天冲刺跑不出来怎么办？
见第 10 节"红黄绿三种结果"。红灯就立刻全员复盘半天，重新审视 ICP / 行业选择 / Prompt 工程， 甚至切换垂直行业。不带感情，看数据说话。不允许"再坚持发 50 封"的自我感动。 21 天的好处是亏损可控 — 即使全输了，3 周时间和几千块成本，团队学到的 Agent 编排 + GBRAIN 上手能力比这值钱。
合规风险 — 爬 LinkedIn / 群发邮件会不会有法律问题？
① LinkedIn 数据用 Apollo / Crunchbase 等合规第三方 API，不直接爬； ② 邮件发送遵守 CAN-SPAM Act / GDPR，每封信都有 unsubscribe 链接，发件方信息真实可验； ③ 不收集 / 储存欧盟个人敏感数据，目标客户主要是企业邮箱（B2B），不是个人邮箱。 Phase 2 会专门做一次合规审查。
AI 模型在涨价 / 在变 — 我们的成本和能力会不会失控？
Helicone 实时监控每个 Agent 的 token 消耗和成本。 Writer / Closer 用 Opus（贵但写信质量重要），其他用 Sonnet / Haiku 路由。 模型迭代是对我们有利 — 同样能力的 LLM 每年成本下降 50%-70%， 2026 年的 Sonnet 比 2024 年的 GPT-4 还便宜还强。
为什么要用 GBRAIN，不自己拼 Postgres + Neo4j？
三个原因：① GBRAIN 是 LLM-WIKI 模式的生产级实现，由 Garry Tan（YC 主席）2026 年 4 月 9 日 MIT 开源，自己生产环境跑着 17,888 页 / 4,383 人 / 723 公司，评测 R@5 达 97.9%； ② 覆盖我们 90% 的需求：客户档案 + 决策人画像 + Deal 追踪 + 知识图谱 + 21 个 cron job 自动维护，全部开箱即用； ③ 自己拼至少省 4-6 周，且 GBRAIN 的"Brain vs Memory"纪律、MECE 归档、Two-Layer Pages（合成事实 + append-only 时间线）这些设计是 Garry 用 1.7 万页生产数据打磨过的，我们短期内不可能想到。 风险点是它给"个人 brain"设计，多租户场景要 fork 加层 —— 这是 Phase 3 的事情，MVP 不影响。
客户的数据隐私 / 安全怎么保证？
GBRAIN 默认 local-first 架构 —— Brain Repo 是本地 git 仓库，PGLite 是嵌入式数据库（无需服务器、无需 Docker、无网络调用），原始数据不出本地。 敏感信息（API key / token / 邮箱地址）在 ingest 时自动 redact。 Phase 2 做产品化时，每个客户公司一份独立 Brain Repo（不混用），从架构上隔离数据。
这件事最坏的失败结局是什么？
最坏：跑了 6 个月发现 AI 开发信回复率到不了 5%，业务员觉得不如手动 — 那么我们至少 ① 自己的储能业务用上了部分 AI 工具，效率提升一截； ② 团队学到了 Agent 编排 + LLM 工程的真本事； ③ 沉淀的 Prompt 库和数据可以转作其他用途。 不是 0 收益的失败，是"昂贵但有用的学习"。这是自己投钱做项目最大的好处 — 输得起。
